# ReAct 代理架构设计

## 1. 概述
本文档概述了课程问答系统中的 ReAct（推理与行动）代理架构。目标是使系统能够执行复杂的推理循环（“思考 -> 行动 -> 观察”），而不仅仅是简单的单轮检索。

## 2. 核心组件

### 2.1 代理核心 (`ReActAgent`)
负责管理对话循环的中央协调器。
- **职责**:
  - 管理对话历史（短期记忆）。
  - 调用 LLM 并提供当前上下文和可用工具。
  - 解析 LLM 输出为“思维”和“行动”。
  - 通过 `ToolExecutor` 执行行动。
  - 处理观察结果并将其反馈给 LLM。
  - 强制执行循环限制（最大迭代次数 = 5-8）。
- **特性**:
  - OpenAI Function Calling standard.
  - Streaming event support (WebSocket).
  - History management.

### 2.2 工具系统 (兼容 MCP)
所有能力的标准化接口。
- **BaseTool**: 抽象类定义：
  - `name`: 唯一标识符。
  - `description`: 供 LLM 理解的自然语言描述。
  - `args_schema`: 用于参数验证的 Pydantic 模型（JSON Schema）。
  - `run`: 异步执行方法。
- **已实现的工具**:
  - `LocalKnowledgeTool`: 增强 RAG（分解 + 多路检索）。
  - `WebSearchTool`: 互联网搜索 (Tavily/DDG)。
  - `CalculatorTool`: 数学运算。
  - `TimeTool`: 获取当前时间。

### 2.3 增强检索 (智能 RAG)
在 `LocalKnowledgeTool` 中实现，用于处理复杂查询。
- **分解**: 将复杂查询（例如，“比较 A 和 B”）分解为子查询（“定义 A”，“定义 B”）。
- **多路检索**: 针对 `TreeStructureRetriever` 并行执行子查询。
- **聚合**: 在返回给代理之前对结果进行重排序和合并。

### 2.4 监控与评估
- **指标**:
  - `tool_usage_count`: 每个工具的计数器。
  - `agent_loop_iterations`: 直方图。
  - `response_latency`: 直方图。
  - `success_rate`: 成功最终答案与错误的比例。
- **追踪**: 记录每一步（思考、行动、观察）并带有唯一的 `trace_id`。

## 3. 数据流

1. **用户查询**: 通过 API 接收。
2. **初始化**: 代理使用系统提示 + 历史记录 + 查询进行初始化。
3. **循环开始**:
    a. **思考**: LLM 生成一个思维（例如，“我需要搜索 X”）。
    b. **行动**: LLM 生成一个工具调用（例如，`search(query="X")`）。
    c. **执行**: 系统执行 `search("X")` 并获得结果 `观察: "X 是..."`。
    d. **更新**: 上下文更新为 思维 + 行动 + 观察。
    e. **重复**: 转到步骤 3a，直到出现“最终答案”或达到最大迭代次数。
4. **最终响应**: 代理将最终答案返回给用户。

## 4. 安全与控制
- **最大迭代次数**: 硬性限制为 8 步，以防止无限循环。
- **超时**: 每个请求的全局超时为 30秒。
- **错误处理**: 如果工具失败，错误将作为观察结果反馈给 LLM（“Error: ...”），允许其重试或尝试不同的策略。
