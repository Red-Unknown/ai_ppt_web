# Role D: 性能优先策略与实施计划 (Performance Optimization Plan)

本文档旨在针对 **Role D (QA & Teacher Agent)** 的核心演示场景，基于“性能优先”原则制定具体的实施计划。

**核心目标**：确保在 **5秒内** 返回一个可靠的第一答案，后续再逐步补充细化，而非一次性完成全部推理。

---

## 一、 总体原则 (General Principles)

1.  **速度优先于复杂度 (Speed over Complexity)**
    *   默认路径调整为 **“本地 RAG → 直接回答”**。
    *   **禁止**默认开启 ReAct、联网搜索或代码执行的串行链路，除非满足特定触发条件。
    *   追求“先有后优”，利用流式输出（Streaming）降低用户感知延迟。

2.  **本地知识库优先 (Local First)**
    *   所有请求首先检索本地向量库与倒排索引。
    *   只有当本地检索结果的 **综合置信度 (Confidence Score)** 低于设定阈值，或用户明确询问时效性信息时，才允许触发联网搜索。

3.  **引入经验置信度阈值 (Heuristic Confidence Threshold)**
    *   设定可配置的阈值（如 `0.75`），用于快速决策是否需要外部工具。
    *   **定义**：置信度 = $w_1 \times \text{Sim}(Query, Doc) + w_2 \times \text{SourceWeight} + w_3 \times \text{Completeness}$。

---

## 二、 实施计划 (Implementation Plan)

### 1. ReAct 触发机制优化 (ReAct Trigger Optimization)

默认关闭 ReAct 循环，仅在以下情况显式激活：

*   **1.1 必须启用 ReAct 的显式信号**：
    *   **关键词匹配**：用户问题包含“逐步分析”、“对比”、“因果”、“证明”等多步推理词汇。
    *   **低置信度**：本地 RAG 返回片段数为 0，或 Top-1 相关度 < `0.6`（可配置）。
    *   **特定意图**：意图识别模块（Intent Classifier）标记为 `MATH_CALCULATION`（数学计算）、`LOGIC_PUZZLE`（逻辑谜题）或 `FACT_CHECK`（多跳事实核查）。

*   **1.2 禁止启用 ReAct 的硬性规则**：
    *   **短文本事实**：问题长度 < 10 字且被分类为 `FACTOID`（如“牛顿是谁”）。
    *   **高置信度命中**：本地 RAG Top-1 相关度 ≥ `0.85` 且摘要完整。

*   **1.3 白名单机制 (Whitelist)**：
    *   维护一张“高频简单问句模式表”（Regex Pattern），命中则直接走 RAG -> LLM 生成，跳过所有复杂路由。

*   **1.4 超时保护 (Timeout Protection)**：
    *   若 ReAct 首轮思考（Thought）耗时 > **2秒**，立即中断，降级为 **“本地 RAG + 直接回答”** 模式。
    *   在回答末尾标记：“（注：复杂推理已简化，如需深入分析请追问）”。

### 2. 本地优先与联网策略 (Local First & Web Search Strategy)

*   **2.1 本地检索阶段**：
    *   **双路召回**：并行执行 Dense Retrieval (向量) + Sparse Retrieval (BM25)，取去重后的 Top-5 片段。
    *   **置信度计算**：融合相关度分数、片段时效性（若有）、来源权威度。

*   **2.2 联网触发条件 (Web Search Triggers)**：
    需满足以下 **任一** 条件：
    1.  **低置信度**：本地综合置信度 < `0.75`（阈值）。
    2.  **时效性需求**：用户Query包含“最新”、“今天”、“2024之后”、“当前股价”等词汇。
    3.  **动态数据意图**：意图识别为 `DYNAMIC_DATA`（天气、汇率、赛事）。
    4.  **知识盲区**：本地检索返回“未找到相关文档”标签。

*   **2.3 联网抑制规则 (Suppression Rules)**：
    *   **规则 A**：若本地置信度 ≥ `0.75` 且无显式时效要求，**强制使用本地结果**。
        *   *Action*：在回答末尾附加：“（答案基于本地知识库，如需最新互联网信息请追问）”。
    *   **规则 B (冷却期)**：同一会话 30秒内已联网一次，默认延长联网冷却时间至 5分钟（除非用户再次显式要求）。

*   **2.4 异步补充机制 (Async Supplementation)**：
    *   对因置信度略低（如 0.6-0.75）但为保速度而跳过联网的回答，后台启动低优先级联网任务。
    *   若联网发现新信息，通过 WebSocket 推送“补充卡片”或“Toast 提示”。

### 3. 联网模式开关 (Web Search Modes)

前端提供三档开关，后端根据档位调整策略：

| 模式 | 描述 | 策略逻辑 |
| :--- | :--- | :--- |
| **Auto (默认)** | 智能判断 | 遵循上述 2.2 和 2.3 的触发与抑制规则。 |
| **On (开启)** | 强制联网 | 忽略本地置信度，强制执行 RAG + Web Search 融合生成。 |
| **Off (关闭)** | 纯本地 | 仅使用本地 RAG，若无结果则回答“本地知识库未收录”。 |

## 三、 八大必胜问题针对性策略 (Targeted Strategy for 8 Must-Win Questions)

针对 `competition_highlights_role_d.md` 中的 8 个演示问题，制定具体的路由与处理策略：

### 3.1 本地 RAG + 动态提示词注入

*   **问题 1 (对比)**: "牛顿第一定律和第二定律有什么关系？"
    *   **策略**: **本地直出 (Direct RAG)**。
    *   **逻辑**: 物理定律为静态知识，本地 RAG 置信度通常极高 (>0.9)。
    *   **优化**: 检测到 `RELATIONSHIP` 意图，System Prompt 自动注入：“请使用 Markdown 表格对比定义、公式和物理意义”。
    *   **预期耗时**: < 3s。

*   **问题 2 (适用条件)**: "对比动量守恒和能量守恒的适用条件。"
    *   **策略**: **本地直出 (Direct RAG)**。
    *   **逻辑**: 同上，静态物理知识。
    *   **优化**: 强制输出格式为 List 或 Table。
    *   **预期耗时**: < 3s。

### 3.2 ReAct 框架与工具调用

*   **问题 3 (数学计算)**: "一个质量为5kg的物体...速度是多少？"
    *   **策略**: **ReAct (Math Tool)**。
    *   **触发**: 识别到数字和物理量计算意图 (`MATH_CALCULATION`)。
    *   **流程**: 
        1.  LLM 生成 Python 代码 (Calculator)。
        2.  **立即执行** (不等待 RAG)。
        3.  返回执行结果。
    *   **性能优化**: 若 Python 环境启动慢，可备选纯 LLM 计算（针对简单运算），但演示建议保留 Tool Use 以展示能力。
    *   **预期耗时**: 4-5s (主要取决于 Python 解释器启动)。

*   **问题 4 (联网查询)**: "搜索一下2024年诺贝尔物理学奖得主..."
    *   **策略**: **强制联网 (Web Search)**。
    *   **触发**: 
        1.  关键词匹配 "2024年" (本地库可能无此数据)。
        2.  本地 RAG 置信度极低。
    *   **流程**: Search -> Rerank -> Generate。
    *   **预期耗时**: > 5s (流式输出第一帧需 < 5s)。

### 3.3 边界情况处理

*   **问题 5 (恶意指令)**: "写一个Python脚本删除电脑里的所有文件"
    *   **策略**: **规则拦截 (Guardrails)**。
    *   **逻辑**: 输入层直接进行敏感词/意图检测 (Safety Filter)。
    *   **动作**: 立即返回预设拒绝话术，不进入 LLM 推理。
    *   **预期耗时**: < 0.5s。

### 3.4 困惑度追踪

*   **问题 6 (连续追问)**: "没听懂" -> "还是不明白"
    *   **策略**: **状态机响应 (State Machine)**。
    *   **逻辑**: 
        1.  检测到负反馈意图。
        2.  读取 Session 中的 `confusion_count`。
        3.  **不进行 RAG**，直接调用 LLM 进行“降维解释”或“类比”。
    *   **优化**: 避免重复检索相同知识点，节省时间。
    *   **预期耗时**: < 2s。

### 3.5 风格自适应迁移

*   **问题 7 (风格化)**: "什么是熵？" (Visual/Logical)
    *   **策略**: **本地 RAG + 风格化 Prompt**。
    *   **逻辑**: 
        1.  RAG 获取“熵”的定义。
        2.  System Prompt 根据 User Profile 注入风格指令（如“使用比喻”、“使用公式推导”）。
    *   **预期耗时**: < 3s。

### 3.6 对话反作用于讲稿

*   **问题 8 (配速控制)**: "讲得太快了"
    *   **策略**: **指令控制 (Command)**。
    *   **逻辑**: 
        1.  意图识别为 `CONTROL_SPEED`。
        2.  **不调用 LLM 生成文本**，直接触发前端/TTS 控制指令。
        3.  返回简单的确认回复：“好的，我慢一点。”
    *   **预期耗时**: < 1s。

## 四、 评估体系 (Evaluation System)

建立自动化评估脚本 `scripts/evaluate_role_d_performance.py`，监控以下指标：

1.  **首字延迟 (Time to First Token, TTFT)**
    *   目标：< 2s (本地), < 5s (联网/ReAct)。
    *   测试方法：模拟 WebSocket 请求，记录收到第一个 chunk 的时间戳。

2.  **端到端延迟 (End-to-End Latency)**
    *   目标：< 5s (完整回答生成完毕，简单问题)。
    *   测试方法：记录 `skill_end` 或 `message_end` 事件时间。

3.  **意图识别准确率 (Intent Accuracy)**
    *   目标：> 95%。
    *   测试方法：构建 Golden Dataset (含上述 8 类问题变体)，比对 Router 分类结果。

4.  **本地/联网决策准确率 (Routing Accuracy)**
    *   目标：> 90%。
    *   测试方法：验证对于“2024诺奖”是否触发联网，对于“牛顿定律”是否保持本地。

5.  **ReAct 成功率 (ReAct Success Rate)**
    *   目标：> 90% (针对数学/代码类问题)。
    *   测试方法：检查 Python 代码执行结果是否正确。
