# ReAct 架构实施报告

## 1. 概述
本文档详细介绍了智能课程助手的“思考 -> 工具 -> 再思考”（ReAct）循环架构的实施情况。该系统支持多步推理、动态工具使用和增强的检索策略。

## 2. 核心组件

### 2.1 ReAct 代理 (`backend/app/services/qa/agent.py`)
- **角色**: 协调推理循环。
- **机制**:
  1.  接收用户查询。
  2.  **思考 (Think)**: 调用 LLM 决定下一步行动（思维）。
  3.  **行动 (Act)**: 如果生成了工具调用，执行工具（例如：知识库搜索、数学求解器）。
  4.  **观察 (Observe)**: 将工具输出反馈给 LLM。
  5.  **循环 (Loop)**: 重复直到得出最终答案或达到最大迭代次数（8次）。
- **特性**:
  - 支持 OpenAI 函数调用标准。
  - 支持 WebSocket 流式传输。
  - 历史记录管理。

### 2.2 增强检索工具 (`backend/app/services/qa/skills/local_rag_skill.py`)
- **角色**: 将本地 RAG 流程封装为可调用的工具 (`local_knowledge`)。
- **策略**:
  - **分解**: 将复杂查询分解为 2-3 个子查询。
  - **多路检索**: 并行异步检索子查询。
  - **聚合**: 去重并合并结果。
  - **指令注入**: 根据意图动态添加约束（例如，“使用表格进行比较”）。

### 2.3 集成 (`backend/app/services/qa/service.py`)
- **工作流**:
  - 标准 QA 意图 -> ReAct 代理。
  - DeepSeek Reasoner 请求 -> 直接推理绕过（针对 R1 优化）。
  - 控制/反馈意图 -> 专用处理程序。
- **流式传输**:
  - 将内部代理事件 (`thought`, `tool_start`, `tool_end`, `answer`) 映射到前端 WebSocket JSON 消息格式。

## 3. 评估体系计划

### 3.1 指标
| 指标 | 目标 | 方法 |
| :--- | :--- | :--- |
| **答案准确率** | ≥ 85% | 人工审核或 LLM作为裁判 (GPT-4o) 对比黄金标准。 |
| **工具选择正确率** | ≥ 90% | 日志分析 `tool_start` 事件与预期工具的对比。 |
| **响应时间** | ≤ 3s | 记录首字时间 (TTFT) 和总持续时间。 |

### 3.2 测试用例（计划中）
- **事实问答**: “牛顿第一定律的定义是什么？”（预期：`local_knowledge` -> 回答）
- **多步推理**: “比较 A 和 B 的速度，已知……”（预期：`math_solver` 或 分解后的 `local_knowledge`）
- **逻辑推理**: “为什么苹果会掉下来？”（预期：思维链）

## 4. 监控方案

### 4.1 可追溯性
- **会话 ID**: 在所有日志和 Redis 状态中跟踪。
- **事件日志**:
  - `[ReAct] Iteration X`
  - `[Tool] Start/End/Duration`
  - `[Error] Exception Stack`

### 4.2 告警
- **高延迟**: 工具执行 > 5s。
- **循环耗尽**: 达到最大迭代次数仍无答案。
- **错误率**: 5分钟内失败率 > 5%。
